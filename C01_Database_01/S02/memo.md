## 관계형 데이터베이스 vs NoSQL
데이터베이스의 핵심: 데이터를 어떻게 하면 더 효과적이고 더 안전하게 관리할 수 있을까?

- DBMS는 목적과 특성에 따라 여러 종류가 있지만, 최근에는 두 분류로 구분됨
    - 관계형 데이터베이스 관리 시스템
        - 현재 가장 널리 사용되는 데이터베이스 시스템
        - 테이블 간의 관계를 기반으로 데이터를 구조화

    - NoSQL(Not Only SQL) 데이터 관리 시스템
        - 빅데이터와 특정 요구사항에 맞춰 최근 등장한 시스템
        - 관계형 데이터베이스의 한계를 보완

### 관계형 데이터베이스 관리 시스템(RDBMS - Relational Database Management System)
- **테이블 기반**
- 행(Row, 레코드)
- 열(Column, 필드, 속성)

### 왜 RDBMS가 기본인가?
- 정형화된 데이터 관리
- ACID 트랜잭션 보장
- SQL이라는 표준 질의어
- 성숙한 기술과 풍부한 생태계
- 실무에서의 압도적인 사용 빈도

### 그럼 NoSQL은 언제 배우면 좋을까?
- 먼저 RDBMS를 제대로 마스터하고 나서 필요할 때 배우면 됨
- 참고로, 데이터를 관리하는 대부분의 문제는 RDBMS로 충분히 해결할 수 있음!!

### NoSQL 데이터베이스 관리 시스템 (Not Only SQL)
- 관계형 모델을 사용하지 않거나, SQL을 주요 데이터 접근 언어로 사용하지 않는 DBMS들을 통칭함
- RDBMS가 모든 상황에서 최적인 것은 아니기 때문에, 특정 요구사항을 만족하기 위해서 사용함
    - 엄청나게 많은 비정형 데이터 처리
    - 매우 빠른 읽기/쓰기 속도
    - 유연한 데이터 모델

### NoSQL 4대 분류 - 간단 정리
- 키-값 저장소
    - 키 -> 값 한쌍
    - 초고속 단건 읽기/쓰기
    - 세션, 캐시
    - Redis, Memcached

- 문서 DB
    - JSON/BSON 문서
    - 유연한 스키마
    - 상품 카탈로그
    - MongoDB

- 컬럼 패밀리 저장소
    - 열(Column) 그룹
    - 대용량 분산/분석
    - 로그/시계열
    - Cassandra, HBase

- 그래프 DB
    - 노드/엣지
    - 복잡한 관계 탐색
    - 추천/이상 탐지
    - Neo4j

- NoSQL은 RDBMS를 대체한다기보다는 상호 보완적인 관계로 이해하는 것이 좋음

### 실무 이야기
- RDB는 실무에 필요한 대부분의 문제를 해결할 수 있는 만능 해결사
- 서비스가 작을 때는 RDB만 사용하다가 서비스가 점점 커지고, 사용자가 늘어나면서 NoSQL이 꼭 필요한 상황에 맞는 NoSQL을 부분적으로 도입하는 것이 좋음
- 각각의 DB가 모두 깊이가 있고, RDB 하나만 제대로 이해하기도 쉽지 않음
- 처음부터 무리하게 다양한 NoSQL을 함께 사용하면 관리해야 하는 시스템이 점점 늘어나므로 시스템 운영이 매우 어려워짐
- 참고로 사용자가 수백만 명 이상이 되더라도 RDB만으로도 충분히 대부분의 서비스를 운영할 수 있음

## 관계형 데이터베이스 종류

### Oracle
- 매우 강력하고, 기능이 정말 많고, 안정성도 최고 수준인 사용 RDBMS의 대표 주자
- 주로 대기업이나 금융권처럼 아주 높은 수준의 성능과 안정성, 그리고 복잡한 기능을 필요로 하는 곳에서 많이 사용됨
- 단점: 매우 비쌈

### MySQL
- 단점: 복잡한 분석 쿼리나 대규모 트랜잭션 처리 성능은 사용 RDBMS나 PostgreSQL에 비해 다소 부족하다는 평가도 있었으나, 지속적인 성능 개선과 새로운 기능 추가로 많은 부분에서 격차를 줄여나가고 있음

### PostgreSQL
- MySQL과 마찬가지로 매우 훌륭한 오픈 소스 RDBMS
- 기능적으로는 MySQL보다 더 뛰어나다고 평가 받는 부분이 많음
    - 복잡한 쿼리 처리 능력
    - 표준 SQL 준수
    - 확장 기능
- 단점: 아직까지 국내 사용자가 많지 않음

### Microsoft SQL Server
- 단점: 마이크로소프트 기술 스택을 사용하는 환경에서 선택함

### MariaDB
- MySQL의 쌍둥이 동생


### H2 Databases(SQLite)
- 주로 임베디드(내장형) 데이터베이스로 사용함
- 별도의 서버를 띄우는 것이 아니라 애플리케이션 자체에 포함되어 실행되는 가벼운 데이터베이스
- 주로 개발 단계에서 빠른 테스트를 위해 주로 사용함

## 관계형 데이터베이스 핵심 개념


### 핵심
- **표(Table)**
    - 행(row)
    - 열(column)

### 핵심 개변 1: 각 데이터를 유일하게 만들어주는 식별자, 기본 키 (Primary Key)
- 다음을 반드시 지켜야함
    - 고유키: 같은 테이블 내에서 절대 중복되면 안됨
    - Not Null: 반드시 값이 있어ㅑ함
- 다른 정보가 같아도 이걸로 구분을 할 수 있음
- 수정 시 같은 값이 있을 경우, 업데이트를 거부함

### 기본 키가 필수인 이유
- 특정 데이터 하나를 빠르고 정확하게 찾아내고, 수정하고, 삭제하기 위해서 사용함
- 기본 키가 없다면 데이터 바다에서 원하는 정보를 특정하기 쉽지 않음

### 핵심 개념 2: 외래 키(Foreign Key) - 따로 떨어진 표들을 "관계"로 묶는 방법
- 외래 키란, 한 테이블(A)의 열(Column)이 다른 테이블(B)의 기본 키(Primary Key) 값을 참조하는 것
- 참조 당하는 쪽: 부모 테이블 (고객)
- 참조 하는 쪽: 자식 테이블 (주문)
    - 어떤 고객이 주문을 했는지 컬럼에 표현
- 자식이 외래키로 부모를 찾아간다고 생각하면 됨

### 외래 키의 중요한 규칙
- 참조 무결성: 외래 키 열에 있는 값은, 반드시 부모 테이블의 기본 키 값 중 하나이거나, 혹은 비어있어야하함(NULL)

### 왜 외래 키를 사용하는가?
- 데이터의 중복을 막고, 데이터의 일관성을 유지하며, 논리적으로 분리된 데이터들 사이에 **관계**를 맺어주기 위함
